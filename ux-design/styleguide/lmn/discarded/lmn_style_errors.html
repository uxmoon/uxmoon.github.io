errors & alerts

This section explains the difference between an error message and an alert, gives guidelines on where each should be used, and sets out standards for the content and design.

An alert message is used when the site performs differently than expected, and no action is needed from the user.

An error message is used to signal mistakes made by the user or site performance problems that prevent further use of the site or site section. Error messages should give the user the information needed to solve a problem.


Flowchart

Alerts

Description
An alert is used when the site performs differently than expected, and no action is needed from the user.

Message content

  • Terminology to be clear, concise and in the users' language.
  • Should give a clear indication of what has happened.
  • Alerts relating to availability and price increases should always start with 'Unfortunately';. The exception is where there is no longer any availability at all, in which case messages should start with 'Sorry'.

Position of message
Located at the top of the page, just below the page title.

Alert types

1. The user does not have cookies enabled.

When a user has removed items from the basket, or is unable to add items to the basket because cookies are disabled.

Design:

  • Should be clearly differentiated from errors, but consistent in format.
  • Text should always stand out from the rest of the site, so pink should never be used.
  • Use of icons and/or colour could be used to differentiate from errors.
  • Site navigation should be retained.

Content:

  • It looks like you don't have cookies enabled on your computer.  Cookies are small pieces of text that sit on your computer which some areas of this site need so that it can work properly."

2. Price or availability change.

Includes:

  • Price change: Where the price of a product changes during the booking process.
  • Availability change: Where a product becomes unavailable during the booking process.
  • No availability: Where there is no current availability for a searched-for product.

Design:

  • Should be clearly differentiated from errors, but consistent in format.
  • Text should always stand out from the rest of the site, so pink should never be used.
  • Use of icons and/or colour should be used to differentiate from errors.
  • Where a price is given (e.g. to notify of a price change), this should be highlighted so that it stands out within the message.

Content:

  • Price increase: "Sorry, the price of this [product] has gone up by £xx.xx. You can continue with this [product] or choose a new [product]." The link anchors down to the results.
  • Price decrease: "The price of this [product] has gone down by £xx.xx!"
  • No availability: "Unfortunately the [product] you selected is no longer available, but there are more great [products] below."

Location:

  • Located at the top of the page, as current.
  • It is not necessary to repeat alerts further down the page.

3. Invalid search criteria.

Where the user has unknowingly entered invalid search criteria, e.g. more than the allowed number of passengers.

The invalid search alert should be eliminated by preventing users from entering the wrong information, either by a) replacing text fields with dropdown lists or b) dynamically updating search forms to adjust to entered criteria. Where this is not possible, then the same standards apply as Search Form Errors.

Content:

  • An example would be, "Unfortunately we cannot book unaccompanied children online. Please select at least one adult or senior passenger or call xxx to book an unaccompanied minor."

4. Search over-specification.

Where the system cannot find any products exactly matching a user’s specification and returns a broader set of results.

Design:

  • Should be clearly differentiated from errors, but consistent in format.
  • Text should always stand out from the rest of the site, so pink should never be used.
  • Use of icons and/or colour should be used to differentiate from errors.

Content:

  • Unfortunately there are no [product] with that [requirement] available. Instead, we have included a list of all [product] with [all/other] [requirement] that are available.
  • Flights example: "Unfortunately there are no flights with that class of seat available. Instead, we have included a list of all flights with other classes that are available."
  • Hotels example: "Unfortunately there are no hotels with that star rating available. Instead, we have included a list of all hotels with all star ratings that are available."
  • Although the majority of messages will follow this format, there will be some exceptions where small changes will need to be made for grammatical reasons.

Location:

  • Located at the top of the page.
  • It is not necessary to repeat alerts further down the page.

Error messages - user errors

Description

Where the user has inadvertently caused the system to be unable to proceed, and must take action to be able to continue.

Message content:

  • The terminology used in the message should always match that on the page.
  • Terminology to be clear, concise and in the users’ language.
  • Should include a clear statement of the problem.
  • Should explain clearly how to recover/proceed.

Position of message:

  • Most error messages should display at the top of the page just below the page title.
  • Search form errors differ in that messages relating to missing fields should appear dynamically next to that field.

Error types

1. Session time-out

Where the user has left the site idle during a booking process.

Design:

  • Page must be updated with new navigation, both global and local.
  • All text to be visible above the fold.

Content:

  • It looks like this session has timed out. This can happen if you leave the site for a while. You can go back and try again. If that doesn’t work you may need to clear your cookies and cache.
  • Include a Back link.

Location:

  • New page.

2. Form error

Where the user has made a mistake or omitted fields while filling in a form.

The error comprises of two parts. The first is the message alerting the user that an error has occurred and giving exact detail on how to proceed, with a link anchoring to the relevant field if this is below the fold. The second part highlights the relevant field making it easy for the user to see where the problem is.

Design:

  • Text should always stand out from the rest of the site, so pink should never be used.
  • additional text is to be used next to the missing or incorrect field, so the existing field and/or label should change graphically to show the user where the problem is.

Content:

  • First line of text to read, "Oops! There’s a problem with your [field type]. Please enter/select [field]."
  • Example: "Oops! There's a problem with your passenger details. Please enter this passenger's last name."
  • Where users have failed to check the box stating they have read the terms and conditions (or similar) the message should follow this format: "Please tick the box to confirm that you [requirement]"

Location:

  • Top of page, just below the page title.
  • Where there is more than one error, each message should start on a new line.

3. Search form error

When the user has omitted a field when filling out a search form.

Design:

  • Message to be contained within the search form in a removable layer.
  • Must be clearly associated with the relevant field.
  • Must give clear instruction on how to remove the message. (Note that 'Close' should be used rather than the current 'OK'.)
  • Text should always stand out from the rest of the site, so pink should never be used.

Content:

There are two possible messages:

  • Where users have left a free text field blank the message should follow this format: "Oops! We need a bit more information before we can start your search. Please [enter] your [missing field]."
  • Where users have not selected an option from a dropdown menu the message should follow this format: "Oops! We need a bit more information before we can start your search. Please select a [missing field] from the list."

Location:

  • Message to appear next to and be clearly associated with the field in question.

Site performance errors

Description

When the site is either fully or partially down, and there is no action the user can take to continue.

Message content:

  • Terminology to be clear, concise and in the users’ language.
  • Should include a clear statement of the problem.
  • Should explain clearly how/when to recover/proceed, including a phone number to the call centre.
  • No debugging or other technical information should be presented on the page.

Position of message:

This type of message should be presented in its own dedicated page.

Site performance error types

1. Platform / database error.

Use when the site is partially down (e.g. the Frontier platform, hotel database).

Design:

  • This is a new page.
  • All text to be visible above the fold.

Content:

  • "The [category] section of the site is currently down. We are working on fixing it and it should be back up again soon. We’re sorry for the inconvenience caused."
  • Include relevant category phone number and link to Contact Us section.

Location:

  • New page.

2. Page cannot be found.

When a page is temporarily unavailable, or has been removed (e.g. if the user has bookmarked an old page).

Design:

  • This is a new page.
  • All text to be visible above the fold.

Content:

  • "We’re sorry. The page you have requested does not exist on our site."
  • Include link to site home page.

Location:

  • New page.

3. Come back later.

To be used when the site is completely offline, for maintenance or due to a technical fault.

Design:

  • This is a new page.
  • Navigation should be removed.

All text to be visible above the fold.

Content:

  • "We’re sorry, but the site is experiencing some problems and is currently down. We’re working on fixing it and it should be back up again soon."
  • Include all category phone numbers.

Location:

  • New page.
